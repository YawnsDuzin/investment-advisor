## 구현 중 발견 및 대응

### 이슈 1: 위치 기반 parity가 도메인 분할과 충돌

초기 plan은 선택자 위치까지 기준으로 parity를 검증하려 했으나, Task 3에서 Mobile Menu Button(원본 라인 1182)을 `03_layout.css`(초기 번들)로 옮기는 것이 strict parity를 깨는 것이 확인됨. 도메인 기반 파일 재편성과 원본 cascade 순서는 근본적으로 상충.

**대응**: Task 4부터 **set-based parity**로 전환 — 동일 선택자·속성 해시·개수는 보존, 위치 변화는 허용. CSS 의미론적으로도 서로 충돌하지 않는 선택자 간 순서 재배열은 안전하므로 이 정책이 타당.

### 이슈 2: `00_legacy.css`가 알파벳순으로 번들 첫 번째

Python `sorted()`로 글롭 결과가 정렬되면 `00_legacy.css`가 `01_tokens.css`보다 앞에 옴. 그러면 legacy에 남은 규칙이 새 파일들보다 먼저 로드되어 cascade 파괴. **대응**: `tools/build_css.py`에 임시 워크어라운드 — `00_legacy.css`를 항상 마지막에 번들. Task 10에서 legacy 삭제 후 이 특수 처리도 제거하여 순수 `sorted()` 복원.

### 이슈 3: `12_dashboard_rest.css`가 500줄 초과

대시보드 메인 영역(Signals, News, Top Picks 3가지 형태, Track Record 위젯) 스타일이 응집성 있게 한 파일에 묶이면서 686줄. 스펙의 500줄 목표를 초과하지만, 추가 분할 시 "대시보드 헤더가 아닌 대시보드의 나머지"라는 일관된 경계가 흐려질 우려. **결정**: 유지. 추후 `12_top_picks.css`, `12_track_record_widget.css` 등으로 세분화 고려 가능(별도 스프린트).
