# OAuth(Google + Kakao) 운영 세팅 가이드

설계 spec: [20260525144745_oauth-google-kakao-design.md](20260525144745_oauth-google-kakao-design.md)

이 문서는 코드 작업 외 인프라/콘솔 등록 절차를 다룬다. **OAuth·Cloudflare 처음 해보는 사람을 대상으로** 개념 설명부터 단계별로 정리한다.

---

## 0. 사전 지식 — 개념 잡기

### 0.1 OAuth 란?

> "다른 서비스(Google/Kakao)에 가입돼 있는 사용자의 신원을, 비밀번호를 우리한테 넘기지 않고 안전하게 확인하는 방법."

기존 방식과 비교:

| 방식 | 사용자 행동 | 우리 서버가 받는 것 |
|---|---|---|
| **이메일 가입** | 이메일·비밀번호 입력 | `email` + `bcrypt(password)` — 비번 해시를 직접 보관 |
| **OAuth (Google/Kakao)** | "Google로 로그인" 버튼 클릭 → Google 동의 화면 → "허용" | Google이 발급한 일회용 `code` → 그걸로 user info 조회 (`sub`, `email`, `name`) |

**핵심**: 비밀번호는 우리 서버를 거치지 않는다. Google·Kakao 가 "이 사용자는 본인이 맞다"고 보증서를 발급해주는 구조. 우리는 그 보증서(=`access_token`)로 사용자 정보만 조회한다.

### 0.2 OAuth 의 흐름 (Authorization Code Flow)

```
[사용자]                [우리 서버]              [Google]
   │                        │                      │
   │── "Google로 로그인" ──▶│                      │
   │                        │── redirect to ─────▶│
   │                        │   Google 동의화면    │
   │◀───────── 동의 화면 보여줌 ────────────────────│
   │                                               │
   │── "허용" 클릭 ────────────────────────────────▶│
   │                                               │
   │◀── redirect to /callback?code=ABC123 ────────│
   │                        │                      │
   │── /callback?code=ABC123 ▶│                    │
   │                        │── code 보내고 ─────▶│
   │                        │   token 교환         │
   │                        │◀── access_token ────│
   │                        │── userinfo 조회 ───▶│
   │                        │◀── {sub, email, …} ──│
   │                        │                      │
   │                        │ DB 조회/생성         │
   │                        │ JWT 쿠키 발급        │
   │◀── 로그인 완료 ────────│                      │
```

용어:
- **`code`**: Google이 "이 사용자가 동의했다"는 일회용 영수증. 30초~10분 정도만 유효, 한 번만 사용 가능.
- **`access_token`**: code 를 교환해서 받는 진짜 토큰. Google API 호출용.
- **`state`**: CSRF 방지용 무작위 문자열. 우리가 발급해서 redirect 에 끼워 보내고, callback 에서 일치 확인.
- **`redirect_uri`**: Google이 "동의 후 어디로 사용자를 돌려보낼까" — **우리 서버의 콜백 URL**. 콘솔에 사전 등록 필수.
- **`scope`**: 요청 권한 범위. 우리는 `openid email profile` 만 요청 (이메일·이름 조회).
- **`sub`**: provider 내부의 사용자 고유 ID. 이메일은 바뀔 수 있지만 `sub` 는 영구. 이걸로 DB 매칭.

### 0.3 왜 HTTPS + 도메인이 필수인가?

Google/Kakao OAuth 는 보안상 다음을 요구한다:

1. **redirect_uri 는 사전 등록된 URL 이어야 함** — 콘솔에 등록 안 한 URL 로 콜백 보내면 거부 (`redirect_uri_mismatch`)
2. **운영 환경은 HTTPS 만 허용** — http://는 `localhost` 만 예외
3. **raw IP 주소는 거부** — `http://1.2.3.4:8000` 같은 형태는 Google이 "Hosts cannot be raw IP addresses" 에러 반환

→ 즉, 라파(라즈베리파이) 같은 홈서버에서 운영하려면:
- 공인 IP 만으로는 안 됨 (raw IP 거부)
- 도메인이 있어야 하고 (예: `myapp.example.com`)
- 그 도메인이 HTTPS 로 우리 서버까지 닿아야 함

여기서 Cloudflare Tunnel 이 등장한다.

### 0.4 Cloudflare 란? Tunnel 이란?

**Cloudflare** 는 글로벌 CDN/보안 회사. 그 중 우리가 쓸 기능이 **Cloudflare Tunnel** (옛 이름: Argo Tunnel).

> "내 라파의 `localhost:8000` 을 Cloudflare 가 관리하는 도메인(HTTPS)에 안전하게 연결해주는 무료 서비스."

기존 방식 vs Tunnel:

| 항목 | 공유기 포트포워딩 | Cloudflare Tunnel |
|---|---|---|
| 공인 IP 노출 | ✓ 노출됨 | ✗ 안 노출 (Cloudflare 가 가림) |
| HTTPS 인증서 | Let's Encrypt 직접 발급·갱신 | ✓ Cloudflare 자동 |
| 포트포워딩 설정 | 공유기 80/443 포워딩 필요 | ✗ 불필요 (outbound 만 사용) |
| 동적 IP 대응 | DDNS 추가 필요 | ✓ Cloudflare 가 자동 추적 |
| 비용 | 무료 (DDNS는 별도) | 무료 |
| DDoS 보호 | 없음 | ✓ Cloudflare 가 흡수 |

**작동 방식**:
```
[인터넷]                [Cloudflare]              [라파]
   │                        │                      │
   │── HTTPS 요청 ────────▶│                      │
   │   myapp.example.com    │                      │
   │                        │◀── outbound 터널 ────│ cloudflared 데몬이
   │                        │   (라파→CF, 항상 유지) │ 라파에서 CF로 먼저 연결
   │                        │── 요청 forward ────▶│
   │                        │◀── 응답 ─────────────│
   │◀── HTTPS 응답 ─────────│                      │
```

라파에서 Cloudflare 로 **밖으로 나가는** 연결(outbound)만 사용하므로 공유기 인바운드 포트를 열 필요가 없다. 보안상 큰 장점.

**도메인이 없다면?** Cloudflare 에서 도메인을 살 수 있다 (`.com` 연 $10 정도). 또는 무료 옵션으로 [DuckDNS](https://www.duckdns.org/) 같은 DDNS 서비스 + Cloudflare 조합도 가능하지만, 깔끔한 건 도메인 하나 사서 Cloudflare 네임서버에 옮기는 것.

### 0.5 우리 시스템에서 OAuth 가 어떻게 동작하는지

설계 spec(`20260525144745_oauth-google-kakao-design.md`) 의 §3 참조. 핵심 차이점:

- **기존**: `users` 테이블에 `email + password_hash` 저장
- **OAuth 추가**: `user_oauth_accounts` 테이블 추가. `(provider, provider_user_id) → user_id` 매핑. 한 user 가 Google + Kakao + 이메일 동시 보유 가능 (1:N)
- **자동 연결**: provider 이메일이 검증됨(`email_verified=true`)이고 동일 이메일 local 계정이 존재하면, 신규 가입 없이 기존 계정에 자동 연결
- **권한 강제 다운그레이드**: OAuth 신규 가입은 무조건 `role='user' / tier='free' / password_hash=NULL`. provider 응답을 그대로 신뢰하지 않음 (권한 상승 차단)

## 1. Cloudflare Tunnel 셋업 (처음부터)

### 1.0 사전 준비 — 도메인 + Cloudflare 계정

1. **Cloudflare 계정 생성**: https://dash.cloudflare.com/sign-up — 무료
2. **도메인 준비**:
   - 이미 도메인 있음 → Cloudflare 에 추가 (`Add a site`) → 안내대로 도메인 등록업체에서 네임서버를 Cloudflare 것으로 변경 → 보통 1~24시간 내 적용
   - 도메인 없음 → Cloudflare 에서 직접 구매 (Domain Registration 메뉴) 또는 가비아·후이즈 등에서 사고 위 절차로 이전
3. **네임서버 이전 완료 확인**: Cloudflare 대시보드에서 도메인 상태가 `Active` 로 표시되면 OK

> Tip: 서브도메인은 자동 인식된다. `example.com` 을 Cloudflare 에 등록하면 `myapp.example.com` 도 자동 사용 가능.

### 1.1 cloudflared 설치 (라파)

`cloudflared` 는 라파에서 Cloudflare 로 outbound 터널을 유지하는 데몬이다.

```bash
# arm64 (라파 4 64bit OS 기본)
curl -L --output cloudflared.deb https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-arm64.deb
sudo dpkg -i cloudflared.deb
cloudflared --version
```

라파 32bit OS 면 `cloudflared-linux-arm.deb`, x86_64 데스크탑이면 `cloudflared-linux-amd64.deb` 로 교체.

### 1.2 Cloudflare 계정 로그인

```bash
cloudflared tunnel login
```

→ 라파에서 브라우저가 열리거나 URL이 출력된다. 그 URL 을 **PC 브라우저로 열어서** Cloudflare 계정 로그인 → 사용할 도메인 선택 → "Authorize" 클릭.

성공 시 `~/.cloudflared/cert.pem` 파일이 생성됨 (이게 Cloudflare API 호출 자격증명).

### 1.3 터널 생성

```bash
cloudflared tunnel create investment-advisor
```

출력 예:
```
Created tunnel investment-advisor with id <tunnel-id>
```

`<tunnel-id>` 는 UUID 형식. 메모해둘 것. `~/.cloudflared/<tunnel-id>.json` 파일도 생성됨 (터널 credentials).

### 1.4 DNS 라우팅 등록

도메인이 우리 터널을 가리키도록 Cloudflare DNS 에 CNAME 자동 등록:

```bash
cloudflared tunnel route dns investment-advisor myapp.example.com
```

→ Cloudflare 대시보드의 DNS 탭에 `myapp` CNAME → `<tunnel-id>.cfargotunnel.com` 자동 추가됨.

### 1.5 config.yml 작성

`~/.cloudflared/config.yml` (또는 `/etc/cloudflared/config.yml`):

```yaml
tunnel: <tunnel-id>
credentials-file: /home/dzp/.cloudflared/<tunnel-id>.json

ingress:
  - hostname: myapp.example.com
    service: http://localhost:8000
  - service: http_status:404
```

- `ingress` 의 위에서부터 매칭. 마지막 항목은 catch-all (필수).
- `service` 는 라파 내부의 백엔드 주소. FastAPI 가 `localhost:8000` 으로 떠 있으면 그대로.

### 1.6 systemd 등록 (상시 실행)

```bash
sudo cloudflared service install
sudo systemctl enable --now cloudflared
sudo systemctl status cloudflared
```

`active (running)` 상태면 OK.

### 1.7 접속 테스트

브라우저에서 `https://myapp.example.com` 접속 → 우리 FastAPI 가 응답하면 성공.

HTTPS 인증서는 Cloudflare 가 자동 발급/갱신한다. 라파에서 `certbot` 같은 거 안 만져도 됨.

### 1.8 트러블슈팅 (Tunnel)

| 증상 | 원인 | 해결 |
|---|---|---|
| `502 Bad Gateway` | FastAPI 가 안 떠있음 | `sudo systemctl start investment-advisor-api.service` |
| `530` (Cloudflare) | 터널 자체가 끊김 | `sudo systemctl restart cloudflared`, 로그 `journalctl -u cloudflared -f` |
| `1033` (Argo Tunnel error) | DNS CNAME 누락 | `cloudflared tunnel route dns ...` 다시 실행 |
| 도메인 접속 안 됨 | 네임서버 이전 미완료 | Cloudflare 대시보드에서 도메인 `Active` 상태 확인 |

## 2. Google OAuth 등록

### 2.1 개념 정리

Google OAuth 를 쓰려면 **Google Cloud Console** 에서 우리 앱을 등록하고 Client ID/Secret 을 받아야 한다.

- **Client ID**: 우리 앱의 공개 식별자. 브라우저 URL 에 노출 OK.
- **Client Secret**: 우리 서버만 알아야 하는 비밀키. token 교환 시 사용. **절대 git/프론트엔드에 노출 금지**.

### 2.2 콘솔 등록 단계별

1. **프로젝트 생성**: https://console.cloud.google.com 접속 → 상단 프로젝트 드롭다운 → "새 프로젝트" → 이름 입력 (예: `investment-advisor`)
2. **OAuth 동의 화면 설정**:
   - 좌측 메뉴 "APIs & Services" → "OAuth consent screen"
   - User Type: **External** 선택 (개인용. Workspace 계정이면 Internal도 가능하지만 일반 사용자는 External)
   - App name: 사용자에게 보일 앱 이름 (예: `Investment Advisor`)
   - User support email: 본인 Gmail
   - App logo: 선택사항. 운영 시 등록 권장.
   - Developer contact information: 본인 이메일
   - "Save and continue"
3. **Scopes 추가**: "Add or remove scopes" → 다음 3개 체크:
   - `.../auth/userinfo.email`
   - `.../auth/userinfo.profile`
   - `openid`
   - → "Update" → "Save and continue"
4. **Test users 등록** (앱 게시 상태가 `Testing` 인 동안):
   - 본인 + 테스트 사용자 Gmail 추가
   - Testing 상태에서는 등록되지 않은 Google 계정으로 로그인 시도 시 거부됨
   - 운영 안정화 후 "Publish app" 으로 전환하면 누구나 사용 가능 (단 민감 scope 추가 시 Google 검증 필요. 우리는 `email/profile/openid` 만 쓰므로 검증 불필요)
5. **OAuth 클라이언트 ID 생성**:
   - 좌측 메뉴 "APIs & Services" → "Credentials"
   - 상단 "+ CREATE CREDENTIALS" → "OAuth client ID"
   - Application type: **Web application**
   - Name: `investment-advisor-web`
   - **Authorized JavaScript origins**:
     - `http://localhost:8000`
     - `https://myapp.example.com`
   - **Authorized redirect URIs** (가장 중요):
     - `http://localhost:8000/auth/oauth/google/callback`
     - `https://myapp.example.com/auth/oauth/google/callback`
   - "CREATE"
6. **Client ID / Client Secret 메모**: 생성 직후 표시되는 두 값을 `.env` 에 저장
   - `GOOGLE_CLIENT_ID=...`
   - `GOOGLE_CLIENT_SECRET=...`

### 2.3 자주 틀리는 부분

- **Redirect URI 는 character-for-character 일치**: trailing slash 유무, http/https, 포트, 대소문자까지 정확히 동일해야 함. 콘솔 저장 후 반영까지 몇 분 지연될 수 있음.
- **Test users 등록 안 한 계정으로 로그인 시도**: `Error 403: access_denied` — Test users 에 추가하거나 앱을 Publish.
- **Client Secret 노출**: 노출됐다면 즉시 콘솔에서 "Reset secret" 후 `.env` 갱신.

## 3. Kakao OAuth 등록

### 3.1 개념 정리

Kakao 는 **Kakao Developers** 콘솔에서 앱을 만들고 "카카오 로그인" 기능을 활성화한다. Google 과 다른 점:

- **REST API 키 = Client ID**. Client Secret 은 선택사항 (활성화하면 보안 ↑, 우리는 활성화 권장).
- **이메일은 필수 동의** 로 설정해야 우리 가입 정책(이메일 매칭)에 맞음.
- **이메일 필수 동의는 비즈앱 전환 필수**. 개인 비즈앱은 사업자등록 없이 신청 가능, 심사 1~3 영업일.

### 3.2 콘솔 등록 단계별

1. **앱 추가**: https://developers.kakao.com → 우측 상단 "내 애플리케이션" → "애플리케이션 추가하기"
   - 앱 이름: `Investment Advisor`
   - 사업자명: 본인 이름 또는 사업자명
   - 카테고리: `재테크/금융` 등 적절히
2. **플랫폼 등록**: 생성된 앱 클릭 → 좌측 "앱 설정" → "플랫폼" → "Web 플랫폼 등록"
   - 사이트 도메인:
     - `http://localhost:8000`
     - `https://myapp.example.com`
3. **카카오 로그인 활성화**: 좌측 "제품 설정" → "카카오 로그인" → 우측 토글 **ON**
4. **Redirect URI 등록**: 같은 화면 "Redirect URI" → "Redirect URI 등록"
   - `http://localhost:8000/auth/oauth/kakao/callback`
   - `https://myapp.example.com/auth/oauth/kakao/callback`
5. **동의항목 설정**: 좌측 "카카오 로그인" → "동의항목"
   - **닉네임**: 필수 동의 (기본)
   - **카카오계정(이메일)**: **필수 동의** 로 설정 → 저장 시도 시 "비즈앱 전환 후 가능" 안내 나오면 다음 단계로
6. **비즈앱 전환** (이메일 필수 동의 위해 필수):
   - 좌측 "앱 설정" → "비즈앱 신청"
   - 개인 비즈앱: 사업자등록 없이 가능. 본인 신분증 사진 + 간단한 앱 설명 제출
   - 심사 1~3 영업일. 통과되면 다시 동의항목 "카카오계정(이메일)" 필수로 설정·저장 성공
7. **Client Secret 활성화** (권장):
   - 좌측 "카카오 로그인" → "보안" → "Client Secret 코드" → "코드 발급" → "활성화 상태 ON"
   - 발급된 코드를 `.env` 의 `KAKAO_CLIENT_SECRET` 에 저장
8. **앱 키 확인**: 좌측 "앱 설정" → "앱 키"
   - **REST API 키** 메모 → `.env` 의 `KAKAO_CLIENT_ID`
   - (JavaScript 키, Native 키 등은 우리가 안 씀)

### 3.3 자주 틀리는 부분

- **비즈앱 전환 없이 이메일 필수 동의 설정 시도** → 저장 불가. 비즈앱 신청부터.
- **이메일 없는 Kakao 계정 로그인 시도** → 콜백에서 `email_required` 반환. UI 에서 "이메일이 등록된 Kakao 계정으로 다시 시도해주세요" 안내.
- **Redirect URI 미등록 또는 오타** → `KOE006` 에러. 콘솔과 callback URL 비교.
- **앱 검수**: Kakao 도 검수 절차가 있으나 이메일·프로필만 쓰는 기본 scope 는 검수 없이 즉시 운영 가능.

## 4. .env 운영 값 예시

```ini
# OAuth 마스터 스위치 — false 면 라우트 미등록 + UI 버튼 자동 숨김 (회귀 0)
OAUTH_ENABLED=true

# Starlette SessionMiddleware 용. state CSRF 토큰 저장.
# 생성: python -c "import secrets; print(secrets.token_hex(32))"
OAUTH_SESSION_SECRET=<64자 hex>

# Google
GOOGLE_CLIENT_ID=<Google Cloud Console>
GOOGLE_CLIENT_SECRET=<Google Cloud Console>
GOOGLE_REDIRECT_URI=https://myapp.example.com/auth/oauth/google/callback

# Kakao
KAKAO_CLIENT_ID=<Kakao REST API 키>
KAKAO_CLIENT_SECRET=<선택 — Client Secret 활성화 시>
KAKAO_REDIRECT_URI=https://myapp.example.com/auth/oauth/kakao/callback
```

> - `OAUTH_SESSION_SECRET` 빈 값 → startup 경고 로그 후 OAuth 자동 비활성화.
> - 로컬 개발 시 redirect URI 는 `http://localhost:8000/...` 로 .env 를 갈아끼우거나, 콘솔에 양쪽 다 등록해두고 코드가 `request.url_for()` 로 동적 생성.

## 5. 의존성 설치

```bash
pip install authlib httpx itsdangerous
```

- **`authlib`**: OAuth 클라이언트 라이브러리. state 관리·token 교환·userinfo 조회를 한 줄로 처리.
- **`httpx`**: authlib 의 HTTP backend (이미 설치돼 있을 가능성 높음).
- **`itsdangerous`**: Starlette `SessionMiddleware` 의 쿠키 서명용.

`requirements.txt` 에 이미 포함돼 있으면 `pip install -r requirements.txt` 로 충분.

## 7. 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| `redirect_uri_mismatch` (Google) | 콘솔 등록값과 .env 불일치 | trailing slash, http/https, 포트, 대소문자 모두 character-for-character 일치 확인. 콘솔 저장 후 몇 분 지연 가능 |
| `Error 403: access_denied` (Google) | 앱이 Testing 상태 + 로그인 계정이 Test users 미등록 | Test users 추가 또는 앱 Publish |
| `KOE006` (Kakao) | Redirect URI 미등록 또는 오타 | Kakao 콘솔 등록 URI 와 콜백 URL 비교 |
| `KOE205` (Kakao) | Client Secret 활성화됐는데 미전송 또는 불일치 | `.env` 의 `KAKAO_CLIENT_SECRET` 확인 |
| Kakao 이메일 동의 화면 안 나옴 | 비즈앱 미전환 또는 이메일 필수 동의 미설정 | 비즈앱 신청 → 동의항목 "카카오계정(이메일)" 필수로 변경 |
| `email_required` (Kakao 콜백) | Kakao 계정에 이메일 미등록 | 사용자가 다른 Kakao 계정으로 시도하거나 이메일 등록 후 재시도 |
| 로컬에서 OAuth hang | `SessionMiddleware` 미등록 또는 `OAUTH_SESSION_SECRET` 빈 값 | `.env` 확인 + 서버 재시작 |
| 콜백 `state mismatch` | 동일 브라우저 두 탭 동시 로그인 시도 또는 세션 쿠키 손실 | 한 탭으로 재시도, 쿠키 차단 설정 해제 |
| `소셜 로그인 계정은 비밀번호가 없습니다` | OAuth-only 유저가 change-password 접근 | 정상 동작 — 비밀번호 설정 기능은 추후 제공 |
| cloudflared: `502 Bad Gateway` | API 서버가 아직 기동 안 됨 | `sudo systemctl start investment-advisor-api.service` |
| cloudflared: `530` | 터널 자체 끊김 | `sudo systemctl restart cloudflared` |
| 도메인 접속 자체가 안 됨 | DNS 전파 미완료 | Cloudflare 대시보드에서 도메인 `Active` 확인, `dig myapp.example.com` 으로 응답 확인 |
| OAuth 버튼이 로그인 페이지에 안 보임 | `OAUTH_ENABLED=false` 또는 환경변수 빈 값 | `.env` 확인 + 서버 재시작 |
