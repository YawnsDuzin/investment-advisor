## 개선된 프롬프트

```
당신은 FastAPI + PostgreSQL 기반 웹 애플리케이션의 인증/인가 시스템 설계 전문가입니다.

## 배경
investment-advisor는 FastAPI + Jinja2 + PostgreSQL 기반 투자 분석 시스템입니다.
현재 인증 없이 모든 사용자가 전체 기능(대시보드, 테마 분석, 채팅, 관리자 페이지)에 접근 가능합니다.
JWT 기반 인증과 역할 기반 권한 관리(RBAC)를 추가하여 보안을 강화하려 합니다.

## 기술 스택 및 제약
- Backend: FastAPI + Uvicorn, Jinja2 템플릿 (SSR 다크 테마 UI)
- Database: PostgreSQL + psycopg2 (schema_version 기반 자동 마이그레이션)
- 기존 구조: shared/(config, db), api/routes/(pages, sessions, themes, proposals, chat, admin)
- 인증 라이브러리: python-jose[cryptography] (JWT), passlib[bcrypt] (패스워드 해싱) 권장
- DB 연결 방식: 함수 단위로 열고 닫음 (커넥션 풀 미사용), RealDictCursor 사용

## 작업 지시

아래 두 기능의 구현 계획을 단계별로 정리한 기술 문서를 작성해주세요:

### 1. JWT 기반 인증 시스템
- 회원가입 (이메일, 비밀번호, 닉네임)
- 로그인 / 로그아웃
- Access Token + Refresh Token 발급 및 갱신 전략
- 비밀번호 해싱 및 검증 방식
- 토큰 저장 방식 (httpOnly 쿠키 vs localStorage — Jinja2 SSR 환경 고려)

### 2. 역할 기반 권한 관리 (RBAC)
- 역할 정의: Admin, Moderator, User 각 역할의 권한 범위
- 기존 라우트별 접근 권한 매핑 (어떤 라우트에 어떤 역할이 접근 가능한지)
  - 예: /admin/* → Admin만, /chat/* → User 이상, / (대시보드) → 인증된 모든 사용자
- FastAPI Dependency를 활용한 권한 검증 미들웨어 설계
- 최초 Admin 계정 생성 방법 (시드 데이터 또는 CLI 명령)
- 관리자 용 관리기능 포함

### 3. DB 스키마 변경
- 필요한 신규 테이블 (users, refresh_tokens 등) DDL
- 기존 schema_version 마이그레이션 체계에 통합하는 방법
- 기존 테이블과의 관계 (예: chat_sessions에 user_id FK 추가 여부)

### 4. 구현 순서 및 파일 구조
- 단계별 구현 순서 (의존성 고려한 작업 순서)
- 신규 생성 파일과 수정 대상 기존 파일 목록
- 각 파일의 역할과 주요 함수/클래스 시그니처

### 5. 보안 고려사항
- JWT 시크릿 키 관리 (.env)
- 토큰 만료 시간 설정 권장값
- CSRF 방어 (Jinja2 폼 기반 로그인 시)
- 비밀번호 정책 (최소 길이, 복잡도)

## 출력 형식
- 형식: Markdown (.md)
- 저장 경로: `_docs/`
- 파일명: `auth_system_implementation_plan.md`
- 구성: 목차 → 아키텍처 개요 → 각 섹션(1~5) → 참고 라이브러리 및 버전

## 제약 조건
- 기존 코드 패턴(shared/db.py 마이그레이션, config.py 환경변수 관리)을 따릅니다
- 외부 OAuth(구글/카카오 등)는 이번 범위에서 제외합니다
  - 기능 구현은 제외하더라도 설계는 추후 추가를 고려하여 설계
- 프론트엔드는 Jinja2 SSR 기반이므로 SPA용 토큰 관리가 아닌 쿠키 기반 세션 관리를 우선 고려합니다
- 기존 API 엔드포인트의 하위호환성을 유지합니다 (인증 미적용 상태에서도 동작 가능한 전환 기간 고려)
```
