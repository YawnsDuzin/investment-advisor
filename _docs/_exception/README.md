# 분석 파이프라인 예외·장애 관리 대장

분석 배치(`analyzer/`)·웹 API 운영 중 발생한 예외·장애·JSON 파싱 실패 등을
구조화된 형태로 기록하고, 원인 분석 → 수정 내역 → 검증까지 추적하는 폴더.

---

## 사용 방법

### 새 이슈 발생 시

1. 파일명은 `YYYYMMDD_<짧은_영문_slug>.md` (예: `20260422_stage1a_json_parse_failure.md`)
2. 아래 템플릿 섹션을 채워 파일 생성
3. 본 README 하단 "이슈 대장" 표에 한 줄 추가
4. 해결된 후에는 상태를 `✅ 해결됨`으로 업데이트하고 **커밋 해시**를 함께 기록

### 기존 원시 로그 덤프

`202604181600_분석오류.md`, `202604182005_분석오류.md`처럼 타임스탬프 접두사 + `_분석오류.md`
형식의 파일들은 **원시 로그 덤프** (구조화 전 기록). 신규 이슈는 상단의 구조화 템플릿을 사용.

## 자주 발생하는 패턴 (Lessons Learned)

### 1. AI 응답 JSON 파싱 실패
- **증상**: `Invalid control character`, `Unterminated string`, `Expecting property name`
- **원인 클래스**:
  - 출력 토큰 한계 근접 시 모델이 포맷 규율을 놓침 (쪼개진 코드블록·자기주석 삽입)
  - 문자열 값 안에 raw 개행·탭 포함
  - JSON 응답이 중간에 잘림 (타임아웃 or max_tokens)
- **대응 레이어**:
  - **프롬프트**: 출력 규율 섹션 + 분량 상한 명시 (`STAGE1_SYSTEM`의 "출력 형식 엄수" 섹션 참조)
  - **파서**: `_sanitize_json_response()` → `_try_fix_truncated_json()` 2단 복구
  - **아카이빙**: 실패·복구 모두 `ai_query_archive`에 `parse_status`로 기록하여 사후 진단

### 2. 긴 Stage 1-A 응답으로 인한 타임아웃
- `QUERY_TIMEOUT` 환경변수를 900초까지 올려도 출력 분량 자체가 크면 비용·지연 모두 악화
- 근본 대책은 **프롬프트 분량 하향** 또는 **스테이지 분할(1-A1 이슈 / 1-A2 테마)**

### 3. KRX 로그인 실패 연쇄
- `.env`의 `KRX_ID`/`KRX_PW` 미설정 시 pykrx 로그인 실패가 스레드별로 수십 번 반복 출력
- 해외 주식은 yfinance 폴백이 있지만, 로그 가독성 저하
- 대응: `shared/config.py`에서 빈 credential 감지 시 로그인 시도 자체를 스킵하도록 가드
