# Prompt Improve Skill

사용자가 제공한 프롬프트를 프롬프트 엔지니어링 기법을 적용하여 더 명확하고 효과적인 프롬프트로 개선합니다.

## 분석 프레임워크

다음 6가지 차원에서 원본 프롬프트를 평가하고 개선합니다:

| 차원 | 핵심 질문 | 개선 방향 |
|------|-----------|-----------|
| **명확성 (Clarity)** | 지시가 모호하거나 중의적인가? | 구체적인 동사, 정량적 기준 사용 |
| **컨텍스트 (Context)** | 배경·목적·대상 독자가 명시되었는가? | "왜", "누구를 위해", "어떤 시스템에서" 추가 |
| **구조 (Structure)** | 단계·순서·출력 형식이 정의되었는가? | 번호 목록, 섹션 구분, 형식 템플릿 제시 |
| **제약 (Constraints)** | 범위·제한·예외 조건이 명시되었는가? | 포함/제외 항목, 길이·형식 제한 명시 |
| **예시 (Examples)** | 기대 결과를 예시로 보여줄 수 있는가? | 입력→출력 샘플, 올바른/잘못된 예 제시 |
| **역할 (Role)** | AI에게 전문가 역할을 부여하면 유익한가? | "당신은 X 전문가입니다" 형태로 역할 설정 |

## 워크플로우

### 1단계: 원본 프롬프트 분석

원본 프롬프트를 읽고 다음을 파악합니다:
- **핵심 의도**: 사용자가 궁극적으로 원하는 것
- **부족한 부분**: 6가지 차원에서 결여된 요소
- **모호한 표현**: 여러 해석이 가능한 단어·구절
- **암묵적 가정**: 말하지 않았지만 전제하고 있는 조건

### 2단계: 개선 프롬프트 작성

분석 결과를 바탕으로 개선된 프롬프트를 작성합니다:

```
[역할 설정 — 해당 시]
당신은 [전문 분야] 전문가입니다.

[배경/컨텍스트]
[프로젝트/시스템 설명, 기술 스택, 관련 제약]

[명확한 지시]
아래 작업을 수행해주세요:

1. [구체적인 첫 번째 작업]
   - 세부 조건 1
   - 세부 조건 2

2. [구체적인 두 번째 작업]
   - 세부 조건

[출력 형식]
결과물은 다음 형식으로 작성해주세요:
- 형식: [마크다운 / 코드 블록 / 표 등]
- 저장 경로: [지정된 경로가 있는 경우]
- 파일명: [파일명 규칙이 있는 경우]

[제약 조건]
- 포함: [반드시 포함해야 할 항목]
- 제외: [포함하지 말아야 할 항목]
- 참고: [참고해야 할 기존 문서, 코드, 규칙]
```

### 3단계: 결과 출력

아래 형식으로 결과를 출력합니다:

```markdown
## 개선된 프롬프트

[개선된 프롬프트 전체 내용]

---

## 개선 포인트

| 차원 | 원본 문제 | 개선 내용 |
|------|-----------|-----------|
| 명확성 | ... | ... |
| 컨텍스트 | ... | ... |
| 구조 | ... | ... |
| 제약 | ... | ... |
| 예시 | ... | ... |
| 역할 | ... | ... |

> 적용되지 않은 차원은 표에서 생략합니다.
```

## 적용 원칙

**보존**: 원본의 핵심 의도와 요구사항을 변경하지 않습니다.

**최소 추가**: 실제로 필요한 정보만 추가합니다. 불필요한 장식은 프롬프트를 복잡하게 만들 뿐입니다.

**구체성 우선**: "자세하게"보다 "다음 항목을 포함하여"가 더 효과적입니다.

**형식 명시**: 출력물의 형식(파일 형식, 구조, 길이)이 명확할수록 결과물의 일관성이 높아집니다.

**코드베이스 컨텍스트 활용**: 현재 프로젝트(ProSafe)에 관한 프롬프트라면, CLAUDE.md의 아키텍처·규칙·파일 경로 정보를 활용하여 컨텍스트를 보강합니다.

### 개선된 프롬프트
```
당신은 Clean Architecture + MVVM 패턴의 .NET WPF 애플리케이션 분석 전문가입니다.

## 배경
ProSafe는 .NET 10 + WPF 기반 통합 출입관리 시스템으로, PostgreSQL DB를 사용합니다.
분석 대상 코드는 현재 프로젝트(ProSafe/ViewModels/, ProSafe/Services/, ProSafe.Data.PostgreSql/)에 있습니다.

## 작업 지시

아래 3개 테이블의 상태 관리 로직을 분석하여 문서를 작성해주세요:
- `workers` — 근로자 정보 및 동기화 상태
- `device_workers` — 근로자-장비 동기화 상태
- `access_logs` — 출입 이벤트 로그

### 분석 범위

**1. 테이블별 상태 컬럼 및 Enum 정리**
각 테이블에서 상태를 나타내는 컬럼(예: WorkStatus, CardSyncStatus, upload_fail 등)을 모두 열거하고,
각 컬럼에 설정되는 값과 그 의미를 Enum 또는 상수 표 형태로 정리합니다.

**2. 수동 처리 경로 분석**
- `근로자 리스트` 화면(WorkerListViewModel, DataService): 근로자 CRUD → 각 테이블 상태값 변화 흐름
- `동기화 관리` 화면(SyncManagementViewModel): 수동 동기화 버튼 → 각 테이블 상태값 변화 흐름

**3. 자동 처리 경로 분석**
- `동기화 관리` 자동 동기화 사이클(8단계): 각 단계에서 테이블 상태값 조회 조건 및 변경 결과

**4. 이벤트별 상태 전이 매트릭스**
근로자 등록/수정/삭제/장비전송/이벤트수신 등 주요 이벤트 발생 시 세 테이블의 상태값이 어떻게 변하는지 표로 정리합니다.

## 출력 형식
- 형식: Markdown (.md)
- 저장 경로: `D:\Project\ProSafe_20260311\ProSafeApp\ProSafe\_docs\`
- 파일명: `상태관리_테이블정리_YYYYMMDD.md` (오늘 날짜 적용)
- 구성: 목차 → 테이블별 Enum → 수동 경로 → 자동 경로 → 상태 전이 매트릭스

## 제약 조건
- 코드에서 실제 확인된 값만 문서화합니다 (추측 금지)
- 레거시(WinForm/MSSQL) 코드는 분석 제외
- 관련 핵심 파일: DataService.cs, WorkerListViewModel.cs, SyncManagementViewModel.cs, DeviceWorkerRepository.cs
```

### 개선 포인트

| 차원 | 원본 문제 | 개선 내용 |
|------|-----------|-----------|
| 명확성 | "자세하게 정리"의 기준 불명확 | 4개 세부 항목(Enum·수동·자동·전이매트릭스)으로 구체화 |
| 컨텍스트 | 어떤 시스템인지, 어떤 파일을 봐야 하는지 미명시 | 기술 스택, 분석 대상 파일 경로 명시 |
| 구조 | 분석 대상이 혼재 | 번호 목록으로 단계 분리, 수동/자동 경로를 독립 섹션으로 구분 |
| 제약 | 어떤 값을 포함/제외할지 불명확 | "코드에서 확인된 값만", "레거시 제외", 핵심 파일 목록 명시 |
| 역할 | AI 역할 미정의 | Clean Architecture WPF 분석 전문가 역할 부여 |
| 출력 형식 | 파일명 규칙 미명시 | 파일명 패턴, 목차 구성 순서 명시 |
